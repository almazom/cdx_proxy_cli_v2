🐛 Root Cause Analysis

The Problem

All 3 authentication keys were being blacklisted with 403 errors, even though the tokens were valid.

Timeline of Discovery

Symptom: All keys blacklisted with "forbidden" status

Proxy logs showed 403 responses from Cloudflare for all requests

Investigation: Direct API calls work

Testing tokens directly against chatgpt.com returned 200 OK

Discovery: Cloudflare challenge headers

Response included cf-mitigated: challenge header

Root Cause Found: Wrong User-Agent header

Proxy was forwarding curl/8.5.0 instead of codex-cli

Technical Details

What Cloudflare Expected

User-Agent: codex-cli
Origin: https://chatgpt.com
Referer: https://chatgpt.com/
Authorization: Bearer <valid_token>

What Proxy Was Sending

User-Agent: curl/8.5.0  ❌
Origin: (missing)  ❌
Referer: (missing)  ❌
Authorization: Bearer <valid_token> ✓

Why This Happened

  1. Upstream URL missing /backend-api: The proxy's chatgpt_backend flag was False because upstream was https://chatgpt.com instead of https://chatgpt.com/backend-api
  2. setdefault() doesn't override: Even when headers were being set, setdefault() couldn't override the incoming User-Agent that was already copied from the client request
  3. Cloudflare WAF blocks suspicious User-Agents: Requests with curl User-Agent to /backend-api endpoints trigger Cloudflare challenges

✅ Fixes Applied

Fix #1: Upstream URL Configuration

File: ~/.codex/_auths/.env

- CLIPROXY_UPSTREAM=https://chatgpt.com
+ CLIPROXY_UPSTREAM=https://chatgpt.com/backend-api

Impact: Enables chatgpt_backend mode which sets required headers

Fix #2: Header Override Logic

File: src/cdx_proxy_cli_v2/proxy/server.py (line ~414)

- base_headers.setdefault("Origin", "https://chatgpt.com")
+ base_headers["Origin"] = "https://chatgpt.com"

- base_headers.setdefault("Referer", "https://chatgpt.com/")
+ base_headers["Referer"] = "https://chatgpt.com/"

- base_headers.setdefault("User-Agent", "codex-cli")
+ base_headers["User-Agent"] = "codex-cli"

Impact: Forces correct headers regardless of incoming request headers

⚠️ Why Codex Cannot Run Without Proxy

Root Cause: Shell Function Override

The codex command is wrapped by a shell function in ~/.zshrc that automatically sets proxy environment variables every time codex is invoked.

Shell Function in ~/.zshrc

# Codex CLI with auto-proxy
codex() {
  # Auto-start proxy and export env vars to this shell
  eval "$(cdx proxy --print-env-only 2>/dev/null)"
  # Run the real codex with proxy environment set
  command codex "$@"
}

Why unset Doesn't Work

Step 1: User runs unset HTTP_PROXY HTTPS_PROXY

Environment variables are cleared

Step 2: User runs codex exec "hello"

Shell function intercepts the call

Step 3: Function executes eval "$(cdx proxy --print-env-only)"

Proxy environment variables are re-set automatically

Step 4: command codex runs with proxy vars

Real codex binary uses proxy regardless of prior unset

How to Run Codex Without Proxy

Option 1: Use command Directly

command codex exec "hello"

Bypasses the shell function entirely

Option 2: Temporarily Disable Function

unset -f codex  # Remove the function
codex exec "hello"  # Now runs without proxy

Option 3: Use Full Path

/home/pets/.npm-global/bin/codex exec "hello"

Running codex without proxy requires valid direct authentication. If your account has rate limits or the direct API is blocked in your region, you may encounter errors like:
